4. デプロイメント設計

SES業務システムは、モノリシックアーキテクチャを採用し、アプリケーションサーバー上に展開される。設定管理はInfrastructure as Code（IaC）の原則に従って行う。

4.1.1 環境構成

SES業務システムは、以下の環境を用意する：

環境 用途 インフラ構成 データ
開発環境 開発者による機能開発、単体テスト アプリケーションサーバー×1, DB×1 開発用データ（匿名化）
テスト環境 QAチームによる結合テスト、性能テスト アプリケーションサーバー×2, DB×1 テスト用データセット
ステージング環境 本番リリース前の最終検証 アプリケーションサーバー×2, DB冗長構成 本番データのコピー（マスキング済み）
本番環境 実サービス提供 アプリケーションサーバー×4, DBクラスタ 実データ
DR環境 災害復旧用バックアップ環境 アプリケーションサーバー×2, DB（スタンバイ） 本番データのレプリカ

4.1.2 展開方式

モノリシックアプリケーションは、以下の方針で展開される：

  • WARファイル: Spring Bootアプリケーションをビルドし、実行可能なWARファイルとしてパッケージ化
  • ロードバランシング: 複数のアプリケーションサーバーインスタンス間でのロードバランシング
  • スケーリング: 負荷に応じたサーバーインスタンスの水平スケーリング
  • ゼロダウンタイムデプロイ: Blue/Green方式またはローリングアップデートによるデプロイ
  • ヘルスチェック: アプリケーションのヘルスエンドポイントによる健全性検証
  • リソース管理: アプリケーションサーバーのCPU/メモリ割り当て管理

4.1.3 デプロイメントパイプライン

継続的インテグレーション/デリバリーの原則に基づき、以下のCI/CDパイプラインを構築する：

  1. コード変更: 開発者がGitHubリポジトリにコードをプッシュ
  2. ビルド: GitHub Actionsがビルドプロセスを実行
    • 依存関係の解決
    • コードのコンパイル
    • 単体テストの実行
    • 静的コード解析
    • WARファイルのビルド
    • 成果物のスキャン
  3. テスト: 自動テストの実行
    • 統合テスト
    • API契約テスト
    • UIテスト
  4. アーティファクト登録: GitHub Packagesまたはアーティファクトリポジトリに成果物を登録
  5. デプロイメント: GitHub Actionsによる環境への自動展開
    • デプロイ構成の更新
    • アプリケーションサーバーへのデプロイ
    • ヘルスチェック
  6. モニタリング: デプロイ後の監視と検証

4.2 ネットワーク構成

SES業務システムのネットワーク構成は、セキュリティとスケーラビリティを重視し、以下のように設計する。

4.2.1 ネットワークアーキテクチャ

[ネットワーク構成図]
ここにネットワーク構成図を挿入

ネットワークは以下のセグメントに分割される：

ネットワークゾーン 用途 アクセス制御
DMZ インターネットからアクセス可能な公開サービス（Webサーバー、CDN） WAF、DDos保護、通信暗号化
アプリケーションゾーン アプリケーションサーバー群 内部通信のみ、セグメント間はファイアウォール制御
データベースゾーン データベース、キャッシュサーバー アプリケーションゾーンからのみアクセス可能
管理ゾーン 監視、ログ、CI/CD、運用管理サーバー VPN経由でのみアクセス可能
バックアップゾーン データバックアップ、アーカイブ 厳格なアクセス制御、データ転送の暗号化

4.2.2 ロードバランシング

システムの高可用性と負荷分散のため、以下のロードバランシング構成を採用する：

  • L7ロードバランサー: Nginx + Hardware Load Balancer
    • SSL/TLS終端
    • パスベースルーティング
    • セッションアフィニティ（スティッキーセッション）
    • レートリミット
  • アプリケーションサーバー間の負荷分散:
    • ラウンドロビン方式
    • ヘルスチェックによる障害検知
    • 自動フェイルオーバー

4.2.3 ネットワークポリシー

ファイアウォールとアクセス制御を用いて、システムコンポーネント間の通信を制御する：

  • デフォルトでは外部からの通信を拒否（deny-all）
  • 必要な通信のみを明示的に許可（least privilege principle）
  • ネットワークセグメント間の通信制限
  • 外部通信の制限（egress filtering）

4.3.3 永続化ストレージ

データの永続化には、以下のストレージを使用する：

オブジェクトストレージ ドキュメント、ログ、バックアップデータ 50TB+（拡張可能） クロスリージョンレプリケーション

4.3.4 高可用性設計

モノリシックアプリケーションの高可用性を確保するため、以下の方針を採用する：

  • アプリケーションの冗長化: 複数のアプリケーションサーバーインスタンスで同一のアプリケーションを実行
  • セッション管理: Redisを利用した集中セッション管理によるステートレス設計
  • 障害検知と自動フェイルオーバー: ヘルスチェックによる障害検知と自動切り替え
  • データベース冗長化: PostgreSQLのレプリケーションによるデータ冗長化
  • バックアップと復旧: 定期的なバックアップとリストア手順の整備
